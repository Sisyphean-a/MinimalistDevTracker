# Minimalist Dev Tracker 改造设计重构文档（V2）

## 1. 文档目的

本文件用于在后续对话中作为唯一上下文来源，指导项目继续改造，避免依赖历史聊天记忆。

范围定义为：

1. VS Code 打开期间，统计 `trackedPaths` 下所有进程对文件的修改（包含 VS Code 外部进程）。
2. 保持“本地存储、无远程上传、可解释错误”的现有原则。

## 2. 当前基线（2026-03-12）

当前代码已具备以下基础模块：

1. 运行时编排：`src/core/runtimeTracker.js`
2. 路径归一化：`src/core/pathKey.js`
3. Git 执行封装：`src/core/gitClient.js`
4. 本地日期键：`src/core/dateKey.js`
5. 存储写队列：`src/core/storageWriter.js`
6. 扩展入口装配：`src/extension.js`

已解决过的问题：

1. `context` 未定义导致的运行时错误。
2. 同日并发写 JSON 覆盖。
3. UTC 归档导致日期偏移。
4. 异步错误可见性不足。
5. 路径大小写语义不一致。
6. Git shortstat 解析受本地语言影响。

## 3. 核心问题（本轮改造目标）

当前仍有一个核心缺口：

1. 活动信号主要来自编辑器事件，外部进程直接改文件时，通常不会触发统计会话。

业务期望：

1. 只要 VS Code 进程还在运行，即使外部工具改动 `trackedPaths` 下仓库文件，也应纳入活跃统计。

## 4. 目标与非目标

目标：

1. 在 VS Code 存活期间，实现“编辑器事件 + 文件系统事件”双信号采集。
2. 报表命令执行前先 `flushAll()`，避免“刚改完显示暂无数据”。
3. 配置 `trackedPaths` 变化后自动重建监控，不要求手动重启窗口。
4. 保持现有 JSON 数据结构兼容。

非目标：

1. 不覆盖 VS Code 关闭后的系统级全时段采集。
2. 不引入远程数据库、账号体系、网络上报。
3. 不重写为系统守护进程（该方向另立文档）。

## 5. 方案决策

采用方案：`VS Code 内增强（推荐）`

理由：

1. 改动范围可控，可复用当前模块。
2. 与现有命令面板、配置项、日报视图一致。
3. 实现成本显著低于系统级守护进程。

## 6. 目标架构

### 6.1 关键模块

1. `runtimeTracker`：统一接收活动信号并路由到 `timeTracker`。
2. `fileActivityWatcher`（新增）：管理 `FileSystemWatcher` 生命周期。
3. `pathRegistry/worktreeDiscovery`：提供允许路径集合与 repo 解析。
4. `timeTracker`：会话状态机（已有）。
5. `storage`：会话落盘（已有并发安全）。

### 6.2 信号来源

1. 编辑器信号：
   - `onDidChangeTextDocument`
   - `onDidChangeActiveTextEditor`
   - `onDidChangeTextEditorSelection`
2. 文件系统信号（新增）：
   - `watcher.onDidCreate`
   - `watcher.onDidChange`
   - `watcher.onDidDelete`

### 6.3 数据流

1. 任一信号进入 `runtimeTracker.recordPathActivity(filePath)`。
2. `pathRegistry` 校验路径是否受监控并解析 `repoPath`。
3. `timeTracker.recordActivity(repoPath)` 更新/续命会话。
4. 超时或 commit 时 finalize，`storage.appendSession` 落盘。
5. 打开日报命令时先 `tracker.flushAll()` 再读取最新日文件。

## 7. 设计细节

### 7.1 新增模块：`src/core/fileActivityWatcher.js`

职责：

1. 根据允许根路径创建 watcher 集合。
2. 接收文件事件并回调 `onFileActivity(fsPath)`。
3. 暴露 `dispose()` 与 `rebuild(nextRoots)`。

接口建议：

```js
function createFileActivityWatcher(options) {
  // options:
  // - vscode
  // - roots: string[]
  // - onFileActivity: (fsPath: string) => void
  // - logError: (label: string, error: Error) => void
}
```

关键点：

1. 使用 `new vscode.RelativePattern(vscode.Uri.file(root), '**/*')`。
2. 显式过滤目录：`.git`（避免无效噪声），其余过滤项通过配置控制。
3. 不吞错，错误统一上报 `logError`。

### 7.2 runtimeTracker 扩展

新增方法：

1. `recordPathActivity(fsPath)`：供文件系统事件直接调用。

保留方法：

1. `recordEditorActivity(document)`。
2. `registerRepository(...)`。
3. `handleCommit(repoPath)`。

### 7.3 extension 装配变更

`activate` 中新增：

1. 创建 `fileActivityWatcher` 实例。
2. 订阅 `workspace.onDidChangeConfiguration`：
   - 若 `minimalTracker.trackedPaths` 变化，重建 path registry + watchers。
3. `openDailyReport` 命令执行流程改为：
   - `await tracker.flushAll()`
   - `await showDailyReport(storage)`

### 7.4 配置项建议（新增）

1. `minimalTracker.fileWatch.excludeGlobs`（array[string], 默认空）  
   用于显式配置额外排除模式，不做隐藏降级。

2. `minimalTracker.flushBeforeReport`（boolean, 默认 true）  
   允许用户关闭“打开报表前 flush”行为，默认开启。

## 8. 测试设计

新增测试文件：

1. `test/fileActivityWatcher.test.js`
2. `test/runtimeTracker.fileActivity.test.js`
3. `test/extension.reportFlush.test.js`

## 9. 分阶段实施计划

### 阶段 A：基础能力接入

1. 新建 `fileActivityWatcher` 及单测。
2. 在 `extension` 装配 watcher，但不启用配置热重载。
3. 验证外部进程改文件可触发会话。

### 阶段 B：体验完善

1. 报表命令前 `flushAll`。
2. 补齐对应测试与错误日志上下文。

### 阶段 C：配置热重载

1. 监听 `trackedPaths` 变更并重建 registry/watchers。
2. 验证重建前后均可正常统计。

## 10. 验收标准

满足以下全部条件视为改造完成：

1. VS Code 打开项目 A，外部进程修改被追踪项目 B 的文件，B 在日报中产生会话数据。
2. 触发 `Minimal Tracker: Open Daily Report` 时，不再频繁出现“刚修改完却暂无统计数据”。
3. 修改 `trackedPaths` 后无需 Reload Window 即生效。
4. 全量测试通过，且无新增静默降级逻辑。

## 11. 手工验证脚本（最小闭环）

1. 启动扩展开发宿主并保持 VS Code 打开。
2. `trackedPaths` 中包含 `F:\Github\MinimalistDevTracker` 与另一个仓库路径。
3. 在终端或其他编辑器修改该另一个仓库下任意文件并保存。
4. 触发一次 commit 或等待会话超时。
5. 执行命令 `Minimal Tracker: Open Daily Report`。
6. 期望看到对应仓库会话数据，不再是“暂无统计数据”。
